# Architecture

**Analysis Date:** 2026-03-13

## Pattern Overview

**Overall:** Static Site Generation (SSG) with Documentation-as-Code

Noble Notations is a **Docusaurus-based documentation site** that generates a static website from markdown content. The architecture follows the Docusaurus framework's opinionated structure with content separated from configuration and styling.

**Key Characteristics:**
- Content-driven architecture (markdown is the source of truth)
- Build-time rendering (no runtime server logic)
- Component-based theming (React components override UI)
- File-system routing (directory structure maps to URL routes)
- Metadata-driven organization (frontmatter controls layout and navigation)

## Layers

**Content Layer:**
- Purpose: Source documents and data for the site
- Location: `/home/ryan/repos/Personal/noble-notations/docs`
- Contains: Markdown files (.md) with YAML frontmatter, category configurations
- Depends on: None (raw content)

- Location: `/home/ryan/repos/Personal/noble-notations/docusaurus.config.ts`, `/home/ryan/repos/Personal/noble-notations/sidebars.ts`

- Location: `/home/ryan/repos/Personal/noble-notations/src/css/custom.css`

- Used by: Rendered HTML pages in browser

**Static Assets Layer:**
- Purpose: Images, logos, and media files served as-is
- Location: `/home/ryan/repos/Personal/noble-notations/static`
- Contains: SVG logos, PNG images, favicon files
- Depends on: None

**Build/Dev Tools Layer:**
- Purpose: Development server, build tooling, and transpilation
- Location: Babel config at `/home/ryan/repos/Personal/noble-notations/babel.config.js`, Node.js ecosystem

- Used by: Docusaurus CLI commands

## Data Flow

**Development Mode:**
1. Developer runs `pnpm start`
2. Docusaurus dev server starts (hot-reload enabled)
3. Reads `docusaurus.config.ts` and `sidebars.ts`
4. Scans `/docs` directory for markdown files
5. Parses frontmatter metadata (title, sidebar_position, slug, etc.)
6. Builds in-memory representation of site structure
7. Renders React components with markdown content
8. Browser accesses http://localhost:3000 and displays live site

**Production Build:**
1. Developer runs `pnpm build`
2. Docusaurus reads config and content
3. Generates static HTML/CSS/JS into `/build` directory
4. Static files are deployment-ready (no server required)
5. Assets optimized for production (minified, cached)

**Deployment:**
1. Built site in `/build` hosted on static file server
2. Site served at https://noble-notations.ryanjnoble.dev
3. URL routing handled by file structure (e.g., `/docs/biltong_logs/batch5.md` → `/biltong_logs/batch5/`)

**State Management:**
- Minimal; Docusaurus handles sidebar state, theme toggle (light/dark), and page navigation
- No client-side application state beyond UI chrome
- Content state is compile-time only

## Key Abstractions

**Documentation Categories:**
- Purpose: Organize content into logical groupings with metadata
- Examples: `/home/ryan/repos/Personal/noble-notations/docs/biltong_logs/`, `/home/ryan/repos/Personal/noble-notations/docs/recipe_tinkering/`
- Pattern: Directory with `_category_.json` file defining label, position, and description; markdown files within category auto-included in sidebar

**Frontmatter Metadata:**
- Purpose: Control how individual pages render and appear in navigation
- Examples: `slug: /`, `sidebar_position: 1`, `sidebar_label: "Biltong notes"`, `title: "Biltong Batch 5"`
- Pattern: YAML block at top of `.md` files; parsed by Docusaurus to override defaults

**Sidebar Configuration:**
- Purpose: Define page hierarchy and navigation structure
- Examples: `sidebars.ts` uses autogenerated sidebars that scan filesystem; manual sidebar definitions possible
- Pattern: TypeScript config file exporting sidebar structure; supports categories, subcategories, and auto-generation

## Error Handling

**Strategy:** Fail-fast at build time for validation errors

**Patterns:**
- `onBrokenLinks: 'throw'` - Build fails if broken internal links detected
- `onBrokenMarkdownLinks: 'warn'` - Warns but doesn't fail on markdown syntax issues
- Frontmatter validation - Invalid YAML in markdown stops processing
- Missing sidebar entries - Docusaurus auto-generates but logs warnings for orphaned docs
- TypeScript checking - `pnpm typecheck` validates config and component types

## Cross-Cutting Concerns

**Logging:**
- Console output during build (warnings/errors from Docusaurus)
- Dev server logs HTTP requests and hot-reload events
- No persistent logging system

**Validation:**
- Markdown frontmatter must be valid YAML
- Internal links must resolve to existing pages
- TypeScript config file must compile without errors
- URL structure validated against slug definitions

**Content Organization:**
- Docs auto-routed based on file structure unless custom slug provided
- Categories defined via `_category_.json` files control sidebar appearance
- Sidebar position metadata controls ordering (`sidebar_position` field)

**Theming:**
- Light/dark mode toggle built into Docusaurus theme
- Colors customizable via CSS variables in `/home/ryan/repos/Personal/noble-notations/src/css/custom.css`
- Primary color scheme: green (#2e8555 light mode, #25c2a0 dark mode)
